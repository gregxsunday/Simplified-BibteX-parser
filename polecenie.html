Kompozycja czy dziedziczenie

Po odkryciu polimorfizmu łatwo jest pomyśleć, iż ponieważ jest on tak sprytnym narzędziem, więc należy stosować dziedziczenie wszędzie. Może to mieć jednak negatywny wpływ na nasze projekty. W istocie wybieranie dziedziczenia jako pierwszego sposobu w sytuacji, gdy wykorzystujemy istniejącą klasę dla stworzenia nowej, powoduje, że sprawy niepotrzebnie się komplikują.

Lepszym wyjściem jest, gdy wybór nie jest oczywisty, wybieranie jako pierwszego przybliżenia kompozycji. Kompozycja nie wymusza budowania projektu jako hierarchii dziedziczenia. Jest także bardziej elastyczna, ponieważ możliwa jest dynamiczna zmiana typów (a przez to zachowania) składowych klasy, podczas gdy w przypadku dziedziczenia dokładny typ musi być znany na etapie kompilacji. //Dynamiczna zmiana zachowania obiektu poprzez użycie kompozycji.
abstract class Actor {
    public abstract void act();
}
 
class HappyActor extends Actor {
    public void act() {
          System.out.println("HappyActor");
    }
}
 
class SadActor extends Actor {
    public void act() {
          System.out.println("SadActor");
    }
}
 
class Stage {
    private Actor actor = new HappyActor();
 
    public void change() {
            actor = new SadActor();
    }
 
    public void performPlay() {
            actor.act();
    }
}
 
public class MainClass {
    public static void main(String[] args) {
       Stage stage = new Stage();
       stage.performPlay(); //Wypisuje "HappyActor"
       stage.change();
       stage.performPlay(); //Wypisuje "SadActor"
    }
}

Obiekt typu Stage zawiera referencję do typu Actor, która jest inicjalizowana na obiekt klasy HappyActor (linia 19). Oznacza to, iż metoda performPlay() działa w pożądany sposób. Ponieważ jednak referencja może w czasie wykonywania zostać związana z innym obiektem, możemy więc podstawić do actor referencję do obiektu SadActor, zmieniając dzięki temu zachowanie metody performPlay(). Zyskujemy zatem dynamiczną elastyczność czasu wykonania. Dla kontrastu: nie możemy zdecydować się na inny sposób dziedziczenia w czasie wykonywania programu - musi to być jednoznacznie określone w chwili kompilacji.

Generalną wskazówką jest: "używaj dziedziczenia dla wyrażania różnic w zachowaniu, zaś pól dla wyrażania zmian stanu". W powyższym przykładzie wykorzystuje się obie techniki: stworzone zostały dwie różne klasy potomne dla wyrażenia różnic w zachowaniu się metody act(), natomiast obiekt Stage wykorzystuje kompozycję dla umożliwienia zmiany swego stanu. W tym przypadku zmiana stanu powoduje zmianę w zachowaniu.

Zadania do wykonania

Modyfikacja sposobu przechowywania elementów

  1. Zaimplementowanie funkcjonalności opisanej w następnym rozdziale wymaga, aby terminy były posortowane, musimy więc zmodyfikować sposób przechowywania zajęć: zamiast mapy LinkedHashMap użyj TreeMap
  2. W przypadku gdy w TreeMap przechowywane są obiekty własnych typów (tu: obiekty klasy Term), a nie wbudowanych, należy, w obrębie danej klasy, nadpisać metodę equals() (zrobiliśmy to na jednych z poprzednich ćwiczeń) oraz skorzystać z jednego z wymienionych poniżej interfejsów.

    LinkedHashMap przechowuje dane w tablicy haszującej. Pobieranie danych z takiej tablicy powinno odbywać się w średnim czasie O(1). Dodanie elementu też, ale jeżeli dodamy element do pełnej tablicy, to musi nastąpić rehashing - musi zostać utworzona druga, większa; następnie są do niej (automatycznie) przenoszone elementy, a stara jest usuwana. A więc o ile nie ustawisz dobrze (albo nie będziesz miał(a) możliwości przewidzieć z góry) wielkości tablicy haszującej, to niektóre dodania mogą mieć złożoność O(n).

    Z kolei TreeMap przechowuje dane w zrównoważonym drzewie binarnym (drzewo czerwono-czarne), a nie w tablicy haszującej, dlatego:

    • pobieranie danych odbywa się w czasie nie większym niż O(log n)
    • metoda hashCode(), zaimplementowana na ostatnich zajęciach, jest zbędna - zamiast niej należy, w zależności od interfejsu, zdefiniować metodę porównującą o określonej nazwie i powinna ona zwracać liczbę całkowitą: ZwracanyWynik = { LiczbaUjemna if ( obiekt 1 < obiekt 2 ) 0 if ( obiekt 1 == obiekt 2 ) LiczbaDodatnia if ( obiekt 1 > obiekt 2 ) }
    Ze względu na lepszą złożoność czasową, należy używać HashMap / LinkedHashMap a nie TreeMap - tej ostatniej używamy tylko wtedy, gdy jest wymagane stałe uporządkowanie kluczy mapy
  3. Zaimplementuj jeden z dwóch poniższych interfejsów:

    Interfejs "Comparable<T>"

    1. Spowoduj aby klasa Term implementowała interfejs Comparable<Term>
    2. Zaimplementuj w niej metodę porównującą compareTo(Term term) - ma porównywać własności obiektu bieżącego (this) z własnościami obiektu wymienionego na liście argumentów; ma zwracać takie wyniki, aby terminy były sortowane rosnąco

    Interfejs "Comparator<T>"

    1. Utwórz klasę TermComparator implementującą interfejs Comparator<Term>
    2. Zaimplementuj w niej metodę porównującą compare(Term term1, Term term2) - ma porównywać własności (dwóch) obiektów wymienionych na liście argumentów; ma zwracać takie wyniki, aby terminy były sortowane rosnąco
    3. Utwórz obiekt klasy TermComparator (utwórz komparator) i przekaż go jako argument konstruktora mapy
    Dla zainteresowanych, proszę przeczytać jakie są istotne różnice (opis 1, opis 2) pomiędzy interfejsem Comparable<T> a Comparator<T>
  4. Sprawdź czy po dodaniu nowej lekcji, klucze mapy są automatycznie sortowane (od 'Pon 8:00' do 'Nie 20:00')
  5. Sprawdź czy po przesunięciu zajęć, klucze mapy są nadal automatycznie sortowane
  6. Spowoduj aby klasa BasicTerm implementowała jeden z dwóch interfejsów: Comparable<BasicTerm> albo Comparator<BasicTerm>
  7. Zaimplementuj odpowiednią metodę porównującą
  8. Za pomocą Arrays.sort() posortuj zawartość tablicy przerw, a następnie wypisz jej zawartość
  9. Rozważ ewentualne korzyści wynikające ze zmiany sposobu przechowywania przerw: zamiast w tablicy statycznej, przerwy są przechowywane w zbiorze TreeSet (przykład)
